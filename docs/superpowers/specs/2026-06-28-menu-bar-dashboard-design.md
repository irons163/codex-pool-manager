# Menu Bar Dashboard Design

Date: 2026-06-28

## Goal

Add a full menu bar dashboard for Codex Pool Manager so users can close the main dashboard window and still see useful pool status from the macOS menu bar.

The first implementation should feel like a complete menu bar utility, but keep the app in normal desktop mode:

- The app remains visible in Dock and Cmd-Tab.
- Closing the main window does not quit the app.
- The menu bar popover continues showing account data.
- The menu bar popover can trigger sync and basic account actions.

This keeps the release risk lower than changing the app into a Dock-less accessory app.

## Non-Goals

- Do not hide the Dock icon in the first version.
- Do not keep the menu bar item alive after the user fully quits the app.
- Do not build a separate login item or helper agent yet.
- Do not recreate every dashboard feature inside the menu bar.
- Do not add detailed cost/model analytics unless the current app data model already has that data.

## Current State

The app already has a minimal `MenuBarExtra` in `CodexPoolManagerApp`.

Current behavior:

- Menu bar title is generated by `MenuBarSnapshotFormatter`.
- Menu bar content uses `MenuBarStatusMenuView`.
- The menu bar model polls `http://127.0.0.1:38477/widget-snapshot`.
- Snapshot data is produced by `WidgetBridgePublisher`.
- The current menu uses `.menuBarExtraStyle(.menu)`, which is good for simple menu rows but too limited for a rich CodexBar-style dashboard.

Main limitation:

The menu bar currently reads a reduced bridge snapshot, not the full app state. This prevents a rich multi-account popover, direct account switching, and reliable background sync when the main dashboard view is not active.

## Proposed Approach

Build a shared app-level pool runtime model and let both the main dashboard and the menu bar read from it.

The menu bar should become a rich SwiftUI popover using `.menuBarExtraStyle(.window)`.

Recommended scope:

- Full menu bar popover dashboard.
- App-level state model shared by dashboard and menu bar.
- Background usage sync owned by the app-level model.
- Menu bar actions for sync, opening dashboard, and switching accounts.
- Keep normal Dock behavior.

## Architecture

### AppPoolRuntimeModel

Create an app-level observable model, owned by `CodexPoolManagerApp`.

Responsibilities:

- Load `AccountPoolState` from `DeveloperAwareAccountPoolStore`.
- Save state changes back to the store.
- Own the usage sync timer.
- Provide derived menu bar summaries.
- Expose safe actions for menu bar and dashboard.

Suggested shape:

```swift
@MainActor
final class AppPoolRuntimeModel: ObservableObject {
    @Published private(set) var state: AccountPoolState
    @Published private(set) var isSyncing: Bool
    @Published private(set) var lastSyncError: String?

    func load()
    func save()
    func syncNow()
    func selectAccount(_ id: UUID)
    func openDashboard()
}
```

The exact names can change during implementation, but the ownership should stay at app level.

### Main Dashboard Integration

`PoolDashboardView` currently owns much of its own state and runtime behavior. The first implementation should avoid a risky full rewrite.

Recommended bridge strategy:

- Add an initializer that can receive an app-level runtime model.
- Keep existing store-based initializer for previews/tests.
- Let the dashboard continue using its existing coordinators.
- Move only the shared state loading, saving, and app-level sync ownership to the runtime model.

This keeps the dashboard stable while making menu bar state reliable.

### Menu Bar Popover

Replace the simple menu with a richer popover:

- Use `.menuBarExtraStyle(.window)`.
- Render `MenuBarDashboardView`.
- Size target: about 360-420 px wide.
- Keep vertical layout compact.

High-level layout:

- Header: app name, active account, last sync time.
- Summary cards: total accounts, available accounts, overall usage, mode.
- Active account section: weekly remaining, five-hour remaining, reset times.
- Account list: all accounts grouped or sorted similarly to the dashboard.
- Warning rows: expired OAuth, excluded sync, relay usage unavailable.
- Actions: sync now, open dashboard, switch account.

The popover does not need to reproduce the full dashboard sidebar or settings.

## Data Flow

Startup:

1. `CodexPoolManagerApp` creates `AppPoolRuntimeModel`.
2. Runtime model loads snapshot through `DeveloperAwareAccountPoolStore`.
3. Runtime model publishes state to `WindowGroup` and `MenuBarExtra`.
4. Widget bridge can still publish a reduced snapshot for widgets.

Usage sync:

1. Runtime model starts the sync timer when the app launches.
2. Sync uses existing `PoolDashboardDataFlowCoordinator` / `CodexUsageSyncService` path where possible.
3. Runtime model merges synced state.
4. Runtime model saves state.
5. Dashboard, menu bar, and widget bridge update from the same state.

Menu bar switch:

1. User clicks an account row action.
2. Runtime model updates selected/manual account state.
3. Runtime model calls the existing switch-launch flow when needed.
4. Runtime model saves state.
5. Dashboard and menu bar refresh immediately.

## Window Behavior

First version:

- Keep `WindowGroup` as a normal desktop window.
- Keep the Dock icon.
- Keep Cmd-Tab visibility.
- Add a reliable "Open Dashboard" action from menu bar.

Implementation detail:

- Prefer a named window group, for example `WindowGroup("Dashboard", id: "dashboard")`.
- The menu bar should use `openWindow(id: "dashboard")` and `NSApp.activate(ignoringOtherApps: true)`.
- This is more reliable than only calling `NSApp.activate`.

Future version:

- Add a setting: "Hide Dock icon when dashboard window is closed."
- That future version can manage `NSApplication.shared.setActivationPolicy(.accessory)` and `.regular`.

## Error Handling

The menu bar should surface account issues without becoming noisy.

Rules:

- Expired OAuth: show a warning row and a "Re-login" or "Open Dashboard" action.
- Relay API key account: show that usage sync is unavailable, not a failure.
- Sync failure: show last sync error in the header area.
- No snapshot/state: show empty state with "Open Dashboard" and "Sync Now".
- Sync in progress: disable duplicate sync button and show progress.

## Localization

All user-facing menu bar strings must use `L10n`.

Required languages:

- English
- Traditional Chinese
- Simplified Chinese
- Japanese
- Korean
- French
- Spanish

Use concise labels because the menu bar popover has limited width.

## Testing

Unit tests:

- Runtime model loads and saves state.
- Runtime model sync updates state and publishes widget snapshot.
- Menu bar summary formatter handles paid and free accounts.
- Menu bar warning model differentiates OAuth expired, relay unavailable, and generic sync failure.
- Account switch action calls the expected switch flow.

View smoke tests:

- Empty menu bar state renders.
- Active paid account renders weekly and five-hour usage.
- Multiple accounts render in compact list.
- Warning row renders for excluded sync.

Regression tests:

- Closing or not showing the dashboard view should not prevent runtime sync.
- "Open Dashboard" should be wired through the named window action.

Manual QA:

- Launch app, close main window, confirm menu bar remains.
- Confirm menu bar still updates after sync.
- Confirm "Open Dashboard" restores the main window.
- Confirm account switching from menu bar matches dashboard state.
- Confirm OAuth expired warning is readable in Traditional Chinese.

## Implementation Slices

1. Introduce `AppPoolRuntimeModel` with load/save and derived summaries.
2. Move app-level sync timer into the runtime model.
3. Wire `CodexPoolManagerApp` to provide the model to both dashboard and menu bar.
4. Convert menu bar to `.window` style and build `MenuBarDashboardView`.
5. Add menu bar actions: sync now, open dashboard, switch account.
6. Add localization keys.
7. Add unit and smoke tests.

Each slice should be separately testable. Avoid combining state ownership refactor and UI polish in one large patch.

## Release Risk

Main risks:

- Moving state ownership may affect existing dashboard behavior.
- Menu bar actions may duplicate dashboard actions if boundaries are unclear.
- Background sync may race with dashboard sync if both remain active.
- Window reopening may be unreliable if only `NSApp.activate` is used.

Mitigations:

- Make the app-level runtime the single owner of background sync.
- Keep existing dashboard coordinators where possible.
- Use a named window group for opening dashboard.
- Add tests around state merge and switch actions.

## Open Decisions

The first version should use these decisions unless changed before implementation:

- Keep Dock icon visible.
- Use full popover dashboard.
- Include background sync in app-level model.
- Include account switching from menu bar.
- Do not add Dock-less accessory mode yet.
